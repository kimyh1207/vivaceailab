# 1-3. 실전 커밋 전략과 메시지 컨벤션

커밋은 저장 버튼이 아닙니다. 코드에 맥락을 부여하는 행위입니다. 좋은 커밋 하나가 수 시간의 디버깅을 막습니다. 나쁜 커밋 하나가 팀 전체를 혼란에 빠뜨립니다.

---

## 커밋을 어떻게 나눌 것인가

가장 흔한 실수는 너무 크게 커밋하는 것입니다.

```
# 나쁜 예
git commit -m "여러 기능 추가 및 버그 수정"
```

이 커밋은 나중에 아무것도 알려주지 않습니다. 어떤 기능인지, 어떤 버그인지, 왜 함께 묶였는지 알 수 없습니다. 되돌려야 할 때도 필요한 것만 골라낼 수 없습니다.

**좋은 커밋의 기준은 하나입니다. 한 커밋은 한 가지 이유로만 변경되어야 합니다.**

```
# 좋은 예
git commit -m "feat: 회원가입 이메일 중복 검사 추가"
git commit -m "fix: 로그인 실패 시 에러 메시지 미노출 버그 수정"
git commit -m "refactor: UserService 메서드 분리"
```

각 커밋이 하나의 이유를 가지면, 히스토리가 곧 문서가 됩니다.

---

## Conventional Commits — 팀의 공통 언어

커밋 메시지에 규칙을 만드는 것이 번거롭게 느껴질 수 있습니다. 하지만 팀이 커질수록, 프로젝트가 길어질수록 규칙 없는 커밋은 소음이 됩니다.

현업에서 가장 널리 쓰이는 규칙은 **Conventional Commits**입니다.

```
<타입>(<범위>): <제목>

<본문 - 선택>

<꼬리말 - 선택>
```

### 타입 종류

| 타입 | 사용 상황 |
|------|-----------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 (코드 변경 없음) |
| `style` | 포맷팅, 세미콜론 등 코드 동작에 영향 없는 변경 |
| `refactor` | 기능 변화 없는 코드 구조 개선 |
| `test` | 테스트 코드 추가·수정 |
| `chore` | 빌드 설정, 패키지 업데이트 등 기타 작업 |
| `perf` | 성능 개선 |

### 실제 예시

```
feat(auth): JWT 기반 로그인 구현

- AccessToken, RefreshToken 분리 발급
- Redis를 활용한 RefreshToken 저장소 구현
- 만료 시 자동 재발급 로직 추가

Closes #42
```

제목만 봐도 무엇을 했는지 알 수 있고, 본문에서 왜·어떻게를 설명하고, 꼬리말로 관련 이슈를 연결합니다.

---

## 커밋 메시지 작성 원칙

**제목은 50자 이내, 명령형으로 씁니다.**

```
# 나쁜 예 — 과거형, 길다
사용자 프로필 이미지 업로드 기능을 추가했습니다.

# 좋은 예 — 명령형, 간결
feat(user): 프로필 이미지 업로드 기능 추가
```

**본문은 "무엇을"이 아니라 "왜"를 씁니다.**

코드를 보면 무엇을 했는지는 알 수 있습니다. 코드를 봐도 알 수 없는 것, 즉 왜 이런 결정을 했는지가 본문에 들어가야 합니다.

```
# 나쁜 본문 — 코드를 읽으면 알 수 있는 것
UserService에 validateEmail 메서드를 추가함

# 좋은 본문 — 코드를 읽어도 알 수 없는 것
이메일 검증 로직을 Controller에서 Service로 이동.
Controller는 요청/응답만 담당해야 하며,
비즈니스 규칙은 Service 계층에서 처리하는 것이 원칙.
```

## 커밋하기 전에 반드시 확인할 것

```bash
# 변경된 파일 전체 확인
git status

# 실제 변경 내용 한 줄씩 검토
git diff

# 스테이징 후 다시 한번 확인
git add -p          # 파일 단위가 아닌 변경 블록 단위로 스테이징
git diff --staged   # 스테이징된 내용 최종 확인
```

`git add -p` 는 특히 중요합니다. 하나의 파일 안에 두 가지 변경이 섞여 있을 때, 블록 단위로 골라서 스테이징할 수 있습니다. 커밋을 깔끔하게 나누는 핵심 도구입니다.

## 커밋 수정이 필요할 때

```bash
# 마지막 커밋 메시지 수정 (아직 push 전)
git commit --amend -m "fix(auth): 오탈자 수정된 커밋 메시지"

# 마지막 커밋에 빠진 파일 추가 (아직 push 전)
git add forgotten-file.java
git commit --amend --no-edit

# 여러 커밋을 정리하고 싶을 때 (아직 push 전)
git rebase -i HEAD~3   # 최근 3개 커밋을 대화형으로 수정
```

**주의:** `--amend`와 `rebase`는 커밋 ID를 바꿉니다. 이미 push한 커밋에 사용하면 팀원들의 히스토리와 충돌합니다. push 전에만 사용하는 것을 원칙으로 합니다.

---

> 커밋 메시지는 미래의 동료에게 보내는 편지입니다. 그 동료가 6개월 뒤의 나일 수도 있습니다.

## 실습

```bash
# 1. 새 파일 생성 후 의도적으로 두 가지 변경을 한 파일에 섞어 만들기
echo "기능 A" >> feature.txt
echo "버그 수정" >> feature.txt

# 2. git add -p 로 블록 단위 스테이징 연습
git add -p feature.txt
# y: 스테이징, n: 건너뜀, s: 더 작게 분할

# 3. Conventional Commits 형식으로 커밋
git commit -m "feat: 기능 A 추가"

# 4. git log 로 히스토리 확인
git log --oneline

# 5. 커밋 메시지 수정 연습
git commit --amend -m "feat(example): 기능 A 추가 (메시지 수정 연습)"
git log --oneline
```

# 1장. 개발 환경과 Git

## 들어가며

코드를 잘 짜는 것만큼 중요한 것이 있습니다. **코드를 안전하게 관리하고, 팀과 함께 일하는 방법**입니다.

아무리 뛰어난 개발자도 개발 환경이 엉켜 있으면 실력을 발휘할 수 없습니다. 반대로 개발 환경과 Git이 손에 익은 개발자는 어떤 팀에 들어가도, 어떤 프로젝트를 만나도 빠르게 적응합니다. 이것이 첫 번째 챕터에서 가장 먼저 이 주제를 다루는 이유입니다.

---

## 왜 개발 환경과 Git이 중요한가

> "내 컴퓨터에서는 됩니다."

개발자라면 한 번쯤 들어봤거나, 직접 해봤을 말입니다. 이 한 마디는 개발 환경이 통일되지 않았을 때 얼마나 많은 시간이 낭비되는지를 단적으로 보여줍니다.

개발 환경 설정은 **한 번 제대로 해두면 평생 자산**이 됩니다. 어떤 프레임워크를 쓰든, 어떤 언어를 배우든 개발 환경을 빠르게 세팅하는 능력은 변하지 않는 기본기입니다.

Git은 그보다 더 근본적입니다. Git 없이 협업은 불가능합니다. Git은 단순한 버전 관리 도구가 아닙니다. **팀의 대화 기록이자, 프로젝트의 역사이자, 실수를 되돌릴 수 있는 안전망**입니다. 커밋 하나하나가 쌓여 프로젝트가 어떻게 성장했는지를 보여주고, 브랜치 전략 하나가 팀 전체의 개발 속도를 결정합니다.

## 이 챕터에서 배울 것

| 섹션 | 핵심 내용 |
|------|-----------|
| 1-1. 개발자의 워크플로 | 코드가 아이디어에서 서비스까지 흐르는 전체 그림을 이해합니다 |
| 1-2. Git의 동작 원리 | Git이 파일을 어떻게 저장하는지, 브랜치가 무엇인지 원리부터 잡습니다 |
| 1-3. 커밋 전략과 메시지 컨벤션 | 읽기 좋은 커밋 히스토리가 팀의 생산성을 높이는 이유를 배웁니다 |
| 1-4. 브랜치 전략과 PR 리뷰 | Git Flow와 Trunk-based, 두 전략의 차이와 선택 기준을 익힙니다 |
| 1-5. ★ GitHub Actions로 CI | 반복 작업을 자동화해 팀이 코드에만 집중할 수 있는 환경을 만듭니다 |

---

## 동기부여

> 개발 환경을 세팅하는 30분이, 이후 수백 시간의 삽질을 막아줍니다.

> Git을 모르면 혼자 개발할 수 있습니다. Git을 알면 세상 모든 개발자와 함께 개발할 수 있습니다.

> 커밋 메시지를 성의 있게 쓰는 사람은, 미래의 자신과 동료에게 편지를 쓰는 사람입니다.

이 챕터를 마치면 여러분은 단순히 명령어 몇 개를 외운 것이 아니라, **협업하는 개발자로서의 기반**을 갖추게 됩니다. 앞으로 배울 Spring Boot, AI, RAG, MCP 등 모든 내용은 이 기반 위에서 시작됩니다.

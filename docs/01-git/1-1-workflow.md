# 1-1. 개발자의 워크플로 한눈에 보기

코드는 혼자 쓰지 않습니다. 아이디어에서 시작해 서비스로 나오기까지, 코드는 수많은 손을 거칩니다. 이 흐름을 모르면 팀에서 길을 잃습니다. 알면 어디서든 자기 자리를 찾습니다.

---

## 코드는 어떻게 서비스가 되는가

개발자의 하루는 대략 이렇게 흘러갑니다.

```
아이디어 → 이슈 등록 → 브랜치 생성 → 코드 작성 → 커밋
→ Pull Request → 코드 리뷰 → 병합 → 자동 테스트 → 배포
```

단순해 보이지만, 이 흐름 안에 협업의 모든 것이 담겨 있습니다.

---

## 단계별로 살펴보기

**1. 이슈 등록**
무엇을 만들지 먼저 말합니다. 기능이든 버그든 이슈로 기록하는 순간, 그것은 팀의 약속이 됩니다.

**2. 브랜치 생성**
`main` 브랜치는 항상 안전해야 합니다. 새 작업은 반드시 별도 브랜치에서 시작합니다. 브랜치는 실험실입니다. 실패해도 `main`은 다치지 않습니다.

```bash
git checkout -b feature/42-user-login   # 이슈 번호와 기능을 이름에 담습니다
```

**3. 코드 작성과 커밋**
작게 나눠 자주 커밋합니다. 커밋은 저장이 아니라 **기록**입니다. 나중에 무슨 일이 있었는지 추적할 수 있어야 합니다.

**4. Pull Request와 리뷰**
혼자 완성한 코드는 없습니다. PR은 "이 코드를 팀에 합류시켜도 될까요?" 라는 제안입니다. 리뷰는 지적이 아니라 검증입니다.

**5. 병합과 배포**
리뷰를 통과한 코드만 `main`에 합류합니다. 병합되는 순간 자동화 파이프라인이 작동하고, 테스트를 통과하면 서비스에 반영됩니다.

---

## 혼자 개발할 때도 같은 흐름으로

팀이 없어도 이 흐름은 유효합니다. 이슈로 할 일을 정리하고, 브랜치를 나눠 작업하고, 커밋으로 기록을 남기는 습관은 혼자 할 때 만들어집니다. 팀이 생겼을 때 자연스럽게 녹아듭니다.

> 좋은 워크플로는 팀을 위한 것이 아닙니다. 미래의 나를 위한 것입니다.

---

## 실습

1. GitHub에 새 저장소를 만들고 이슈 하나를 등록해보세요.
2. 이슈에 대응하는 브랜치를 만들고 (`git checkout -b feature/이슈번호`), 파일 하나를 수정한 뒤 커밋해보세요.
3. PR을 열고 직접 리뷰 코멘트를 달아보세요. 혼자라도 이 과정을 한 번 끝까지 해보는 것이 중요합니다.

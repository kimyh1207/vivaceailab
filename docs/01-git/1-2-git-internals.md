# 1-2. Git의 동작 원리(스냅샷·브랜치·HEAD)

Git을 명령어로만 외우면 금방 막힙니다. 모르는 상황이 오면 손이 멈춥니다. 원리를 알면 다릅니다. 처음 보는 에러도, 낯선 명령어도 스스로 풀어낼 수 있습니다.

---

## Git은 파일을 어떻게 저장하는가

많은 사람이 Git을 "변경된 내용만 저장하는 도구"라고 생각합니다. 틀렸습니다.

Git은 **스냅샷(Snapshot)** 을 저장합니다. 커밋할 때마다 그 시점의 프로젝트 전체를 사진 찍듯 저장합니다. 변경되지 않은 파일은 이전 스냅샷을 그대로 참조해 용량을 아끼지만, 개념적으로는 매번 전체 상태를 기록합니다.

```
커밋 A        커밋 B        커밋 C
─────────     ─────────     ─────────
index.html    index.html →  index.html (변경)
style.css  →  style.css     style.css →
main.js    →  main.js    →  main.js

(→ 는 이전 스냅샷 참조, 변경된 파일만 새로 저장)
```

이것이 Git이 빠른 이유입니다. 특정 시점으로 돌아가는 것은 타임머신이 아닙니다. 해당 스냅샷을 꺼내는 것뿐입니다. `git checkout` 한 번으로 수백 개의 파일이 순식간에 바뀌는 것도 같은 이유입니다.

---

## 커밋은 어떻게 연결되는가

커밋은 혼자 존재하지 않습니다. 각 커밋은 자신이 어디서 왔는지, 즉 **부모 커밋**을 기억합니다. 이 연결이 쌓여 히스토리가 됩니다.

```
[커밋 A] ──→ [커밋 B] ──→ [커밋 C]
  최초          두 번째       세 번째
```

각 커밋은 내부적으로 이런 정보를 담고 있습니다.

```
커밋 오브젝트
├── 작성자 (Author)
├── 작성 시각 (Timestamp)
├── 커밋 메시지
├── 스냅샷 참조 (Tree)
└── 부모 커밋 ID (Parent)
```

`git log` 로 보이는 40자리 해시값(`a3f2c1d...`)은 이 정보들을 모두 합쳐 만든 고유 ID입니다. 커밋 내용이 조금이라도 바뀌면 ID가 완전히 달라집니다. 그래서 Git의 히스토리는 변조가 불가능합니다.

---

## 브랜치는 무엇인가

브랜치를 코드의 복사본이라고 생각하는 사람이 많습니다. 실제로는 훨씬 가볍습니다.

브랜치는 **특정 커밋을 가리키는 포인터**, 그 이상도 이하도 아닙니다. `.git/refs/heads/` 폴더 안에 있는 텍스트 파일 하나입니다. 파일 안에는 커밋 ID 40자리가 적혀 있을 뿐입니다.

새 브랜치를 만드는 데 1초도 걸리지 않는 이유가 여기 있습니다. 수백만 줄의 코드를 복사하는 게 아니라, 텍스트 파일 하나를 만드는 것입니다.

```
main     ──→ [커밋 C]
feature  ──→ [커밋 C]   ← 같은 지점에서 출발, 파일 복사 없음
```

feature 브랜치에서 새 커밋을 쌓으면, feature 포인터만 앞으로 이동합니다. main은 그대로입니다.

```
main     ──→ [커밋 C]
                │
feature  ──→ [커밋 C] ──→ [커밋 D] ──→ [커밋 E]
                                          ↑
                                     feature 포인터
```

브랜치가 "실험실"인 이유가 바로 이것입니다. feature에서 무슨 실험을 해도 main은 전혀 영향을 받지 않습니다.

---

## HEAD는 무엇인가

브랜치가 커밋을 가리킨다면, HEAD는 **브랜치를 가리킵니다**. 즉, "지금 내가 어떤 브랜치에서 작업하고 있는가"를 나타내는 포인터입니다.

```
HEAD → main → [커밋 C]
```

`git checkout feature` 를 실행하면 HEAD가 feature로 이동합니다. 작업 디렉터리의 파일들도 feature의 최신 커밋 스냅샷으로 교체됩니다.

```
HEAD → feature → [커밋 E]
```

`git log` 결과에서 `(HEAD -> main)` 이라고 보이는 것, 이제 정확히 읽힙니다. "HEAD가 main을 가리키고 있고, main은 이 커밋을 가리킨다"는 뜻입니다.

### Detached HEAD 상태

가끔 `HEAD detached at a3f2c1d` 라는 메시지를 만납니다. 당황하지 않아도 됩니다.

이것은 HEAD가 브랜치가 아닌 **특정 커밋을 직접 가리키는 상태**입니다. 과거 커밋을 탐색하거나 특정 버전을 확인할 때 발생합니다. 이 상태에서 커밋을 하면 어떤 브랜치에도 속하지 않아 나중에 찾기 어렵습니다. 탐색 후에는 반드시 브랜치로 돌아오세요.

```bash
git checkout main   # 또는 작업 중인 브랜치 이름
```

## 세 개념이 합쳐지면

| 개념 | 실체 | 역할 |
|------|------|------|
| 스냅샷 | 커밋 오브젝트 | 특정 시점의 프로젝트 전체 상태 |
| 브랜치 | 텍스트 파일 (커밋 ID) | 특정 커밋을 가리키는 포인터 |
| HEAD | 텍스트 파일 (브랜치명) | 현재 작업 중인 위치 |

`merge`는 두 브랜치의 스냅샷을 합치는 것입니다. `rebase`는 커밋들의 부모를 바꿔 히스토리를 다시 쓰는 것입니다. `reset`은 브랜치 포인터를 과거 커밋으로 되돌리는 것입니다. 모두 이 세 개념 위에서 작동합니다.

> Git이 어렵다는 느낌은 명령어를 몰라서가 아닙니다. 그 명령어가 내부에서 무엇을 움직이는지 모르기 때문입니다.

## 실습

직접 눈으로 확인하는 것이 가장 빠릅니다.

```bash
# 1. 현재 HEAD와 브랜치 상태를 그래프로 확인
git log --oneline --decorate --graph

# 2. HEAD가 실제로 어떤 파일인지 확인
cat .git/HEAD
# 출력 예시: ref: refs/heads/main

# 3. 브랜치가 실제로 어떤 파일인지 확인
cat .git/refs/heads/main
# 출력 예시: a3f2c1d4e5b6... (40자리 커밋 ID)

# 4. 새 브랜치 생성 후 HEAD 변화 확인
git checkout -b test-branch
cat .git/HEAD
# 출력 예시: ref: refs/heads/test-branch

# 5. 커밋 오브젝트 내부 확인
git cat-file -p HEAD
# 출력 예시: tree, parent, author, committer, 메시지
```

`cat .git/refs/heads/main` 을 실행했을 때 나오는 40자리 숫자, 그게 브랜치의 전부입니다. 이것을 눈으로 확인한 순간, Git은 더 이상 블랙박스가 아닙니다.
